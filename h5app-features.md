作为服务 3 万跑者同时在线的移动端应用，选手端的小程序（或 H5 页面）在产品交互上必须遵循极其严苛的三个原则：免注册、极简交互、气泡高频响应。

跑者在赛前或者赛中状态下，往往处于高心率、手出汗、精神高度紧凑的场景中（例如在起点人挤人时寻找厕所）。因此，界面绝不能有复杂的层级和密密麻麻的文字，应当以 “对话流 + 业务捷径卡片” 为核心进行设计。

以下是选手端应用的详细交互设计方案：

1. 选手端首页视觉与分流区 (The Hub)

打开小程序的第一秒，选手必须在 0.5 秒内找到他想要的功能。



顶置动态通知栏：根据赛事时间轴自动切换。

赛前：“距离2026成都马拉松开跑还有 14 小时，请尽快领取参赛包。”

赛中：“突发：由于前方赛道突降暴雨，请注意脚下防滑，最近补给点在 1.5 公里后。”

四大免提问“金刚区”快捷键：

为了帮 Java 后端和 Dify 减压，最频繁的需求不走 AI，直接点击气泡卡片：

【我的参赛信息】（一键展示：参赛号、分区、专属领物窗口、应急联系人）。

【赛道补给地图】（点击直观展示带有公里数的赛道图）。

【成绩与计时查询】（冲线后一键查询）。

【SOS 紧急求助】（高亮红色，防误触设计）。

2. 核心 AI 交互流设计 (The Conversational AI)

界面中下部是类似 ChatGPT 交互的对话框，但针对跑步场景做了专门的优化。

选手免登录无感绑定逻辑：

选手通常通过扫描号码布上的专属二维码进入。

Java 后端在 URL 中携带加密的 runner_id 或通过微信一键授权手机号。

实现效果：跑者点开直接打招呼：“Hi 张三，你是 A 区 10234 号选手，请于 7:15 前到达 A 区检录。”

输入方式优化：

超大语音输入按钮：针对跑步时手抖、出汗不便打字，提供类似微信的语音转文字输入，Java 接收后调用语音转文字（STT）再送入 Dify API。

快捷联想气泡 (Suggested Prompts)：

对话框上方常驻 3 个动态语义推荐词（根据赛前赛中智能切换），选手一键点击即可提问：

赛前推荐：“我的号码布在哪个柜台领？” “明天几点关门？”

赛中推荐：“下个厕所在几公里？” “医疗点能处理擦伤吗？”

3. RAG 结构化卡片回复 (Structured Reponse Cards)

当选手向 Dify 提问时，如果 AI 仅仅回复一段干巴巴的文字，体验会非常糟糕。Java 后端应该拦截 Dify API 的输出，将其包装为可视化业务卡片。

场景 1：问位置、距离（如“最近的补给点在哪？”）

烂回复：“前方 2 公里有补给站，提供水和香蕉。”

对麦智能卡片回复：

┌────────────────────────────────────────┐
│ 🏪 下一站点：15公里核心补给站 (距离您 450 米)  │
├────────────────────────────────────────┤
│ 💧 物资提供：魔力电解质水、香蕉、盐丸      │
│ 🚑 医疗配置：配备AED、除颤仪及救护车      │
├────────────────────────────────────────┤
│  [ 📍 点击一键唤起高德地图导航前往 ]     │
└────────────────────────────────────────┘



*   **场景 2：问个人参赛情况（如“我应该去哪领物？”）**
    *   *对麦智能卡片回复*：直观弹出包含选手姓名、12 号领物窗口的条形码卡片，志愿者现场扫描即可发包。

## 4. 赛中生命线：SOS 一键救援模块 (Emergency SOS Trigger)
这是控制台联动的“杀手级安全功能”，必须在前端做极简化、防误触交互。

<Sequence>

  <Step subtitle="需长按 3 秒" title="长按触发救援">
    选手在小程序任意界面长按右下角的红色 “SOS” 浮窗 3 秒。长按机制能够完美防止选手在跑步甩手、手机放口袋时发生误触。
  </Step>
  <Step subtitle="1 秒内完成" title="自动捕获 GPS 坐标与环境状态">
    小程序触发手机底层的 GPS 定位权限，并自动读取当前的电量、网络信号强度。
  </Step>
  <Step subtitle="大颗粒按钮选择" title="确认/勾选伤情症状">
    屏幕弹出 3 个巨大的色块按钮：【心脏不适/胸闷】、【肌肉拉伤/抽筋】、【外伤出血/跌倒】。选手若无法点击，5 秒无响应后系统默认按最高级别伤情提交。
  </Step>
  <Step subtitle="Java 异步高可靠队列" title="数据并轨上报">
    Java 后端瞬间将跑者的“参赛号、血型、紧急联系人电话、精准 GPS 坐标、勾选伤情”打包，通过 RabbitMQ 强力推送到组委会控制台大屏，并伴随全屏闪烁报警，同时向赛道最近的医疗志愿者发送定位。
  </Step>
</Sequence>

## 5. 极简技术实现建议（前端开发层面）

为了保证在 2026 年各类高低端手机、甚至选手在赛道上高负荷、低网络信号下依旧能流畅打开：
1.  **采用原生小程序或极轻量 H5**：严禁引入任何重度动效、大体积的 JS 插件。首包体积控制在 1MB 以内，确保在 3G/4G 弱网环境下 2 秒内必须渲染出对话框。
2.  **SSE（流式传输）原生支持**：由于调用的是 Dify API 的流式输出（Stream），前端必须完美支持 Server-Sent Events (SSE) 协议，让 AI 的回答像打字机一样一个字一个字蹦出来，降低选手的焦虑感（因为等待整段回答生成可能需要 5-8 秒）。
